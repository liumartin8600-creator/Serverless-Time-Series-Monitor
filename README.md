## 项目简介

本项目将 QDII 基金盘中估值、历史数据、实时溢价、统计套利、雷达预警、交易记录与 AI 分析集中在一个 Web 终端中。系统通过 Workers 提供后端 API 与定时任务，通过 D1 保存历史数据与交易流水，通过 KV 保存用户设置、缓存、AI Provider 配置与快照数据。

项目目标不是替代人工判断，而是将分散的数据源、估值口径、历史统计和提醒机制整合为一个可复核、可追踪、可自动化读取的研究工具。

## 功能特性

### 盘中监控

- 实时抓取汇率、指数、期货和基金市价。
- 推导盘中实时净值与动态溢价率。
- 支持自动刷新和手动刷新。
- 按资产类别分组展示盘中监控图表。
- 支持雷达卡片、告警、推送和排序管理。

### 实时估值

- 展示动态溢价率排序。
- 展示实时市价、官方净值、盘中涨跌、晚间预估净值和动态溢价。
- 提供估值公式说明，便于核对估值来源和计算口径。

### 深度研究与统计套利

- 支持多基金历史序列比较、差值序列绘图和净值涨跌幅对比。
- 支持静态 / 动态布林带回测、多组标准差网格回测。
- 支持历史触发明细弹窗和配对套利参数矩阵。
- 支持图表联动与触发记录查看。

### 水位 / 极值

- 展示历史极值、时间分位、空间分位和当前位置水位。
- 用于判断当前基金溢价或价差处于历史区间中的相对位置。

### 量化矩阵

- 绝对空间：历史均值、标准差、Z-Score、时间分位。
- 相对估值：同类均值、同类偏离 Spread、Peer Z-Score。
- 动量趋势：7 日 / 30 日均线、DIF、Momentum。
- 综合诊断信号：左侧抄底、右侧持有、高估切换等提示。

### 网格执行

- 推导 S2 / S1 / 均值中枢 / R1 / R2 执行区间。
- 适合 T+0 挂单、分批买卖、区间交易和做 T 执行参考。

### 统计套利 / 多基金轮动

- 支持多基金轮动池构建。
- 支持相对均值回归信号和动态 Z-Score 指令。
- 支持满仓轮转策略全局寻优、轮动图表和动态策略矩阵。

### 实盘战报

- 支持主账户、子账户 1、子账户 2 等多账户交易流水录入。
- 支持交易记录增删改查。
- 展示实盘收益曲线、策略超额收益对比和历史战报矩阵。

### 数据审计

- 支持历史缺失数据增量抓取。
- 支持单基金全历史同步。
- 支持历史数据分页查询与图表展示。
- 支持估算净值修正、批量修正、数据库物理硬复权和 CSV 导出。

### AI 决策

- 支持多模型路由，包括 Google、NVIDIA、自定义代理、ModelScope 等 Provider。
- 支持流式对话输出。
- 支持将全盘量化矩阵自动注入 AI 分析上下文。
- 可用于生成配对套利、高低切换、抄底逃顶和风险提示等辅助分析。

### 估值修正

- 支持日级修正因子管理。
- 支持批量修正。
- 支持估值误差诊断和同一交易日口径核对。

### 系统设置

- 支持自动刷新频率配置。
- 支持推送通道配置。
- 支持雷达配置和其他用户偏好持久化保存。
- 支持 AI Provider 的 API Key 配置，API URL 使用代码内置默认地址。

## 页面模块说明

### 实时大屏

实时大屏用于展示盘中行情、基金估值、动态溢价、宏观因子、板块均值、雷达卡片和实时刷新状态。该页面是盘中观察和雷达触发的主要入口。

### 今日雷达

今日雷达用于展示单基金或双基金价差雷达状态，支持买点 / 卖点阈值、动态 Spread 曲线、曲线悬停查看、图例显隐控制、声音告警和 Webhook 推送。雷达状态可由页面访问或定时任务驱动更新。

### 估值修正

估值修正用于管理和校准估算净值与真实净值之间的偏差。系统会检查同一行数据中的收盘价、实际净值、估算净值、真实溢价、估算溢价和系统误差是否使用统一交易日口径。

### 深度研究

深度研究用于多基金历史对比、价差曲线、布林带回测、标准差网格回测和历史触发明细分析。

### 实盘战报

实盘战报用于记录账户交易流水，计算实际换仓利差、累计收益和策略超额表现。

### 系统设置

系统设置用于配置自动刷新、雷达参数、Webhook 推送通道、AI Provider 和其他用户偏好。相关配置会保存到 KV。

### AI 分析

AI 分析模块会结合盘面数据、估值结果、量化矩阵和历史研究结果，为用户提供辅助分析和风险提示。

### Webhook 推送

Webhook 推送模块支持 Bark、PushDeer、钉钉机器人、飞书机器人等通道，可用于雷达触发通知和通道连通性测试。

## 核心逻辑说明

### 估值链路

QDII 盘中估值推演链路主要包含：

- T-2 官方净值。
- 隔夜标的涨幅。
- 人民币中间价影响。
- 盘中外盘期指影响。
- 离岸人民币 CNH 盘中影响。
- 管理费日损耗近似扣减。
- 用户在估值修正 Tab 中配置的修正因子。

系统会基于上述因子推导 **晚间预估净值**、**盘中实时净值** 与 **动态溢价率**。该链路的核心目标是判断基金当前价格相对估算净值是否合理，而不是只观察基金自身涨跌。

### 溢价率计算

溢价率核心口径为：

```txt
真实溢价 = 收盘价 / 实际净值 - 1
估算溢价 = 收盘价 / 估算净值 - 1
系统误差 = (实际净值 - 估算净值) / 估算净值
```

在估值修正 Tab 中，同一行数据应尽量使用同一个交易日的数据：

- 收盘价：交易日 D 的 A 股收盘价。
- 实际净值：交易日 D 对应的基金公司公布净值。
- 估算净值：系统对交易日 D 净值的最终估算值。
- 真实溢价、估算溢价和系统误差：基于同一行数据现算或按同一口径展示。

### AGENT_SNAPSHOT_V1

`AGENT_SNAPSHOT_V1` 是面向第三方程序和智能体读取的结构化快照，主要包含：

- 宏观行情与汇率数据。
- 今日雷达状态。
- 实时估值明细。
- 生成时间和诊断信息。

该快照可用于 Python 脚本、自动化报告、外部智能体和其他第三方工具读取当前系统状态。

### 今日雷达 Spread 定时点

今日雷达 Spread 定时点用于记录当日雷达组合的价差变化。系统会将相关数据写入 `radar_spread_history`，用于前端曲线展示、阈值对比和盘中状态回放。

### 午间休息保护

北京时间 `11:30:00` 至 `13:00:00` 为午间休息保护窗口。在该窗口内：

- 跳过 `AGENT_SNAPSHOT_V1` 写入。
- 不生成今日雷达 Spread 定时点。
- 覆盖首次打开大屏、手动刷新、轮询接口和定时任务等入口。
- 不只是打印跳过日志，而是真正阻止后续写入动作。

### KV 配置读取

KV 用于保存用户偏好、AI Provider 配置、宏观缓存、估值修正和快照数据。读取 KV 时需要考虑：

- KV 数据为空。
- JSON 损坏或字段缺失。
- 新旧配置结构兼容。
- 未配置 API Key 的 Provider 不应被调用。
- 缓存异常时应尽量安全降级。

## AI Provider 配置说明

系统支持通过 KV 管理 AI Provider 配置，避免在代码中写死真实密钥。

当前支持的 Provider 包括：

- Google
- NVIDIA
- OpenRouter
- ModelScope

系统设置 Tab 中可配置：

- API Key
- API URL 使用代码内置默认配置，不再通过系统设置页面手动填写。

保存配置后，API Key 等相关信息会写入 KV。后续 AI 调用会优先从 KV 读取密钥配置，并使用代码中的默认 Provider URL；未配置 API Key 的 Provider 不会被错误调用。旧版本 KV 中已有的 `URLS` 字段仍保持兼容读取。

> 安全要求：README、源码仓库和日志中不应出现真实 API Key。示例中只应使用 `YOUR_API_KEY` 或“用户自行配置的密钥”等占位说明。

## 数据与缓存说明

### D1 数据库

D1 主要用于保存：

- 基金历史数据。
- 实盘交易流水。
- 历史估值底稿。
- 今日雷达 Spread 历史记录。

### KV

KV 主要用于保存：

- 用户偏好：`USER_CONFIG_PREFS`
- AI Provider 配置：`AI_PROVIDER_CONFIG`
- 当日修正因子：`CORRECT_日期`
- 宏观缓存：`LATEST_MACRO_CACHE`
- 第三方快照：`AGENT_SNAPSHOT_V1`
- 盘中估算净值快照：`EST_NAV_日期`
- 其他临时或调度状态数据

### 缓存与降级

- 当实时行情接口异常时，系统可尝试读取 `LATEST_MACRO_CACHE` 作为兜底。
- AI Provider 配置读取失败时使用安全默认配置。
- KV 数据为空、JSON 损坏或字段缺失时，应避免接口 500，并按安全逻辑降级。

## 使用与部署说明

### 1. 准备环境

确保已安装 Node.js 与 Wrangler CLI：

```bash
npm install -g wrangler
wrangler login
```

### 2. 创建 D1 数据库

```bash
wrangler d1 create fund-history-db
```

记录返回的 `database_name` 和 `database_id`。

### 3. 创建 KV 命名空间

```bash
wrangler kv:namespace create FUND_KV
```

记录返回的 KV `id`。

### 4. 配置 `wrangler.toml`

```toml
name = "qdii-quant-monitor"
main = "worker.js"
compatibility_date = "2024-03-20"

[[d1_databases]]
binding = "DB"
database_name = "fund-history-db"
database_id = "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"

[[kv_namespaces]]
binding = "FUND_KV"
id = "xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"

[triggers]
crons = ["*/3 1-8 * * *", "*/3 0,14 * * *"]

[vars]
DEFAULT_FEE_RATE = "0.008"
```

如果入口文件路径不同，请将 `main` 改为实际路径，例如：

```toml
main = "src/index_new.js"
```

### 5. 注入运行时密钥

```bash
wrangler secret put ADMIN_TOKEN
wrangler secret put GOOGLE_API_KEY
wrangler secret put NVIDIA_API_KEY
wrangler secret put MODELSCOPE_API_KEY
wrangler secret put CUSTOM_API_KEY
```

> 上述命令只表示密钥名称，不应在 README、源码或提交记录中写入真实密钥值。

### 6. 初始化数据库

#### 表 1：基金历史数据

```sql
CREATE TABLE IF NOT EXISTS fund_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    Code TEXT NOT NULL,
    Price_Date TEXT NOT NULL,
    NAV_Date TEXT,
    Market_Price REAL,
    Official_NAV REAL,
    Static_Premium REAL,
    Est_NAV REAL,
    Scrape_Time TEXT,
    UNIQUE(Code, Price_Date)
);
```

#### 表 2：交易流水

```sql
CREATE TABLE IF NOT EXISTS trade_journal (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    account TEXT DEFAULT 'Main',
    trade_date TEXT NOT NULL,
    fund_from TEXT,
    fund_to TEXT,
    qty_from REAL DEFAULT 0,
    price_from REAL DEFAULT 0,
    qty_to REAL DEFAULT 0,
    price_to REAL DEFAULT 0,
    actual_alpha REAL DEFAULT 0,
    note TEXT,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

#### 表 3：持仓数据

```sql
CREATE TABLE IF NOT EXISTS fund_positions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    account TEXT DEFAULT 'Main',
    fund_code TEXT NOT NULL,
    total_qty REAL DEFAULT 0,
    available_qty REAL DEFAULT 0,
    avg_price REAL DEFAULT 0,
    unrealized_pnl REAL DEFAULT 0,
    note TEXT,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

 

### 7. 发布

```bash
wrangler deploy
```

发布完成后，访问分配的项目地址即可使用。

## Cron 与自动化能力

### 已实现内容

- Worker `scheduled()` 定时任务。
- 手动 Cron 调度中心。
- 自动净值更新。
- 盘前 / 盘中逻辑执行。
- 自动快照生成。
- 宏观缓存刷新检查。

### 当前 Cron 时间表

#### 盘前 / 盘中触发

```cron
*/3 1-6 * * *
```

对应北京时间约 `09:00 - 14:59`，每 3 分钟执行一次盘中流程。

#### 非盘中净值更新

```cron
*/3 0,14 * * *
```

对应 UTC `0` 点（北京时间 `08:00-08:59`）和 UTC `14` 点（北京时间 `22:00-22:59`），每 3 分钟执行一次净值更新任务。

### 2. 估值修正链路检查

针对 Tab“估值修正”中系统测算误差偏大的问题，检查了真实净值、估算净值、收盘价、真实溢价、估算溢价和系统误差之间的完整计算关系。

检查重点包括：

- 是否存在 T 日 / T+1 日错位。
- 是否存在北京时间、UTC 时间或交易日日期错位。
- 是否存在估算溢价来自旧缓存，而估算净值来自另一来源。

- 是否存在同一行数据口径不一致。

调整目标是尽量统一估值修正 Tab 中同一行数据的交易日口径，并增加必要诊断能力，便于继续定位误差来源。

## 常见问题

### 为什么估算净值和实际净值可能仍然存在误差？

QDII 基金估值受到海外市场、汇率、期货、基金公司净值发布时间、交易日口径和缓存数据等因素影响。系统会尽量统一口径并提供诊断信息，但估算净值不等同于基金公司最终公布净值。

### 为什么午间休息期间不写入 Spread 定时点？

午间休息期间 A 股市场不连续交易，如果继续写入定时点，容易在今日雷达曲线上产生不具备交易意义的点位。系统在北京时间 `11:30-13:00` 跳过相关写入。

### 为什么 AI Provider 配置放到 KV？

将 API Key 放到 KV 中，可以避免在代码中硬编码真实密钥；Provider API URL 使用代码默认配置，避免因页面误填 URL 导致 AI 调用失败。

### 为什么不建议把密钥写在 README 或源码里？

README 和源码很容易被提交到仓库或同步到外部环境。真实 API Key、Token、Webhook 地址或管理员密码一旦泄露，可能导致账户滥用、推送通道暴露或后台接口被未授权访问。

### 为什么文档中仍保留部署和初始化说明？

部署和初始化说明属于项目使用的必要上下文，但本文档的重点仍是项目功能、核心逻辑、版本更新、代码更新过程和安全注意事项。
